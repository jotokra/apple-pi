Why this, not another wrapper

Other agents hand you a model. This one runs itself.

apple-pi is a refined distribution of the open-source Pi Coding Agent — the way a Linux distro wraps a kernel. The engine is Pi's. apple-pi adds the three things that turn a model into an agent you can trust:

  • It proves its own setup. Onboarding makes one real call to confirm your key works, then destroys the key. No stale secrets left in a file.
  • It tunes itself to your model — reads the provider catalog (context window, thinking format, cost) and rewrites its config to match. Switch models, re-tune.
  • It improves itself, safely. A loop watches how you work, writes a weekly brief of concrete changes, and never applies one without your yes.

Where your keys live

🔐

Your keys don't sit in a dotfile. They don't leak — by design.

After install, every key you add lives in one encrypted vault, and the safe path is the easy path: paste into a masked prompt that shows dots, not glyphs — it never reaches your input line, your session log, or your shell history.

trace-free entry

A masked prompt shows dots as you type — shoulder-surf safe. No plaintext reaches sessions, telemetry, logs, or history. Enforced + smoke-tested, not asserted.

encrypted at rest

One file, locked with a passphrase you choose (aes-256-cbc, mode 0600). /vault list shows names; secrets never print without an explicit reveal.

you gate every reveal

Nothing exports a secret unless you ask. Onboarding's key is marked temporary and pruned the instant the connection is proven; keys you add yourself persist.

yours, anywhere

Export into pi's auth, or pipe to your own manager (1Password CLI, pass) — secret on stdin, metadata via env, never a command-line argument.

Defense in depth against accidental leakage — not a sealed vault against a thief with your laptop and passphrase. Turn on FileVault for the outer wall; the vault is the inner one.

The self-improvement loop

It watches how you work, then asks before it changes anything.

① collect (daily)

Behavioral telemetry from your session logs → a local SQLite store. No LLM calls, zero quota burn.

② brief (weekly)

A diff vs. last week with concrete, reproducible change proposals — tool-use patterns, memory pressure, errors.

③ review → apply

apple-pi review shows the diff; apply --latest --yes writes it. You gate every change.

Voice mode

Type it, or talk it. Same conversation.

Type in the pi TUI, then /voice (or Ctrl+V) flips the session into pivoice — on-device speech-to-text (whisper.cpp) and text-to-speech (say). Your voice turns append to the same JSONL; pi -c brings them back into the tree. One conversation, two ways in.

Beyond the filesystem

The agent can see and act on the live web.

A bundled web extension — on by default — turns “use the best viable way” outward: search the web, read pages, and drive your own persistent, headed Chrome so you watch every click. Logins survive between runs.

web_search

Ranked results (title / url / snippet). DuckDuckGo free; Tavily / Brave via key.

web_fetch

URL → cleaned markdown. Renders JavaScript-heavy SPAs through the browser on demand.

browser_*

Snapshot → click / type / check by ref, screenshot, tabs. The agent confirms before any payment or deletion.

Two channels, kept distinct

Know what's your behavior adapting vs new code arriving.

self-improvement

Proposals derived from your session telemetry — tool-discipline drift, compaction pressure, error rates. Tagged source: autoresearch. You review then apply --yes.

release updates

New apple-pi / pivoice versions from GitHub. update --check is read-only; folded into the weekly brief too. Tagged source: release. Never auto-installs.

you gate both

Nothing writes itself. The brief proposes; you read the diff and say yes — for config and for code.
